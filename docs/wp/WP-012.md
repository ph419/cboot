## Context（为什么做这件事）

外部报告《cboot.ps1 启动报错（一闪而过）问题报告》指出：`cboot.ps1` 若为 UTF-8 **无 BOM** 编码，在 Windows PowerShell 5.1（双击运行 / `cboot.cmd` 默认调用环境）下，中文字符会被按 GBK（CP936）解码，导致引号、大括号配对错乱，触发数十条解析错误，PowerShell 立即退出 → 表现为"窗口一闪而过"。

经验证，**当前项目 `D:\cboot\cboot.ps1` 确实存在此问题**（并非只存在于报告所指的桌面副本）：

| 验证项 | 结果 |
|--------|------|
| `cboot.ps1` 首三字节 | `23 20 3D`（`# =`）→ 无 BOM |
| 非 ASCII（中文）字符数 | 457 处 |
| **Windows PowerShell 5.1 AST 解析** | **37 条错误**（Line 356/357/370/372/381/502/503 等，均为中文相关行）|
| `scripts\test-context-window.ps1`（对照） | `EF BB BF` → 有 BOM，正常 |

**为何开发者无感**：在 PowerShell 7（pwsh，默认按 UTF-8 解码）下脚本可正常运行；但终端用户双击 `cboot.cmd`（走 `powershell.exe` 5.1）时必然"一闪而过"。脚本逻辑/语法本身完全正确，纯粹是编码问题。

## 问题分析（根因）

1. **直接原因**：`cboot.ps1` 文件编码为 UTF-8 无 BOM，文件包含大量中文注释与 `Write-Host` 中文字符串（457 处）。
2. **解码不匹配**：Windows 双击 `.ps1` / `cboot.cmd` 默认调用 `powershell.exe`（**Windows PowerShell 5.1**）。PS 5.1 对无 BOM 的 `.ps1` 按系统 ANSI 代码页（中文系统为 GBK/CP936）解码，而非 UTF-8。
3. **连锁错误**：中文 UTF-8 字节中部分值（如 `0x22`/`0x7D`/`0x29`）恰好对应 ASCII `"`, `}`, `)`。GBK 解码后，字符串终止符与大括号配对被打乱，引发 37 条解析错误，整个脚本根本未能解析成功 → PowerShell 立即抛错退出 → 窗口一闪而过。
4. **结论**：问题不在脚本逻辑，而在文件编码。

## 修复方案

将 `cboot.ps1` 转存为 **UTF-8 with BOM**（首字节 `EF BB BF`）。这是最稳妥的长期方案，同时兼容 PowerShell 5.1 与 7。转换不改变任何内容字节，仅前置 BOM：

```powershell
$src = 'D:\cboot\cboot.ps1'
Copy-Item -LiteralPath $src -Destination "$src.bak"   # 备份原文件
$content = Get-Content -LiteralPath $src -Raw -Encoding UTF8
$utf8Bom = New-Object System.Text.UTF8Encoding($true)
[System.IO.File]::WriteAllText($src, $content, $utf8Bom)
```
