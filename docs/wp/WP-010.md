## Context（为什么做这件事）

用户在 cboot 添加模型时填了上下文窗口 **1M**，但生成的 settings 文件却写入了：
- `"CLAUDE_CODE_DISABLE_1M_CONTEXT": "1"`（不该有——1M 不应禁用 1M 上下文）
- `"CLAUDE_CODE_AUTO_COMPACT_WINDOW": "200000"`（不该有——既非 1M 也非用户输入值，是上次 200K 模型的残留）

**正确行为**：填 1M 时，settings 里不应有 `CLAUDE_CODE_DISABLE_1M_CONTEXT`，也不应有 `CLAUDE_CODE_AUTO_COMPACT_WINDOW`。

## 问题分析（根因）

cboot 中有**三处**写入上下文窗口 env 的逻辑，其中两处一致、一处遗漏：

| 位置 | `< 1M` 分支 | `>= 1M` 分支（清除遗留字段） |
|------|-------------|------------------------------|
| `New-SettingsTemplate` (`cboot.ps1:298-302`) | ✅ 设置 DISABLE_1M_CONTEXT + AUTO_COMPACT_WINDOW | ❌ **缺失（本 WP 要修的 bug）** |
| `Add-Model` 已存在文件分支 (`cboot.ps1:1291-1303`) | ✅ 设置 | ✅ 移除两个字段 |
| `Edit-ModelConfig` (`cboot.ps1:1666-1678`) | ✅ 设置 | ✅ 移除两个字段 |

**触发链**：
1. `New-SettingsTemplate` (`cboot.ps1:206-211`) 会复用 `~/.claude/` 下**第一个**已有的 `settings-*.json` 作为模板
2. 若该模板来自一个 200K 模型，其 `env` 里已含 `CLAUDE_CODE_DISABLE_1M_CONTEXT=1` 与 `CLAUDE_CODE_AUTO_COMPACT_WINDOW="200000"`
3. 用户新建 1M 模型时，`$contextWindowSize -lt 1000000` 为 false，跳过 if 分支，**也没有 else 清除**
4. 模板里的两个字段被原样序列化进新文件 → 出现 `200000` 残留

**`200000` 是关键证据**：它不可能是用户输入 `1M` 解析结果（应为 1000000），只可能是复用的 200K 模板残留。

此外，`$template` 可能是 **hashtable**（硬编码 fallback，`cboot.ps1:226`）或 **PSCustomObject**（从文件 `ConvertFrom-Json`，`cboot.ps1:217`）——参考 `cboot.ps1:306-315` 已有的双类型处理，新逻辑必须同样兼容两种类型。

## 修复方案

抽取统一辅助函数 `Set-ContextWindowEnv`，封装"按窗口大小 set/remove 两个字段 + 兼容两种类型"，三处调用统一改用。既修掉 `New-SettingsTemplate` 的遗漏，又消除三处重复逻辑，防后续遗漏。
